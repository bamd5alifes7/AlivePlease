# 技術筆記撰寫準則

這份準則用來整理 technical notes 類文件的寫作偏好，供其他專案沿用。

## 文件目的

1. 快速知道專案使用哪些技術，以及選擇它們的理由。
2. 快速回顧實際處理過的問題、判斷與結果。

技術筆記不是產品規格、操作手冊或功能清單。功能行為只在解釋技術選擇、問題原因或維護限制時提及；完整流程、欄位格式與使用方式應留在規格或使用文件。

## 口吻

- 使用中性、自然的開發者筆記口吻。
- 不要寫得像第三者替專案解說。
- 假設文件可能由其他工程師閱讀，內容要專業且容易維護。
- 記錄實際做過的事，不刻意包裝技術能力。

## 結構

建議順序：

1. 技術總覽：用表格列出技術、用途與選擇原因，只作為索引。
2. 技術主題：依認證、資料層、部署等脈絡分段，每段一次寫完選擇理由、實作、問題與結果。

不要為了套用固定格式，另外建立「主要實作經驗」「踩過的問題」「技術經驗摘要」。這種切法容易讓同一事件被重複描述。

## 主題歸併原則

- 同一個技術選擇、問題原因、修改與驗證結果，應留在同一節。
- 踩坑紀錄直接放在所屬技術主題下，不另外集中成一章。
- 共用相同背景與結論的內容，應合併而不是換句話重述。
- 只有能獨立理解，且具有不同原因或不同結論的事件，才拆成不同主題。
- 結尾摘要預設省略；只有能提供新的查找價值時才保留。

## 技術選擇的寫法

- 「用途」回答這項技術在專案中做什麼。
- 「選擇原因」回答為什麼使用它，以及它符合哪些專案條件。
- 有明確比較過替代方案時，簡短記錄取捨；沒有比較過就不要補造理由。
- 若只是框架預設或沿用既有技術，也應如實寫明。
- 遇到不熟悉的術語，先用一句白話說明，再保留正式名稱供延伸查找。
- 原因保持簡短；需要數據、限制或替代方案時，再放到所屬技術主題詳述。

例如：

- database session、JWT、`auth()` 延遲與實測結果，屬於同一條身分驗證脈絡。
- Vercel region、DB round trip 與 production latency，屬於同一條部署效能脈絡。
- Prisma transaction 雖然發生在 Vercel，根本原因在資料存取方式，應放在 Prisma 主題。

## 內容取捨

應該保留：

- 實際使用的技術與用途。
- 選擇或更換技術的理由，以及當時的專案條件。
- 有數據支持的效能或行為差異。
- 已發生問題的現象、原因與處理方式。
- 對維護仍有幫助的限制與判斷依據。

應該避免：

- 太長的背景說明。
- 教科書式技術介紹。
- 重述產品功能、畫面流程、設定方式或 API 規格。
- 把 README、SPEC 或操作手冊的內容搬進技術筆記。
- 不會持續維護的改善清單。
- 沒有明確執行意義的 TODO。
- 同一事件散落在多個章節。

## 文字風格

- 優先短句、表格與條列。
- 標題依技術脈絡命名，不依文章功能命名。
- 每個技術點要連到專案中的實際用途。
- 問題紀錄至少說清楚「為什麼發生」與「怎麼解決」。
- 完稿後檢查每個事件是否出現超過一次；若有，優先合併。

## 標題範例

推薦：

- 技術總覽
- Next.js 應用架構
- 身分驗證與 Google 整合
- Supabase 與 Prisma
- Vercel 部署與效能

避免：

- 主要實作經驗
- 踩過的問題
- 技術經驗摘要
- 後續改善清單
- 未來 TODO
